## Что работает прямо сейчас

- **Дашборд**: `http://localhost:8088` (на Mac, через `cd backend && source venv/bin/activate && uvicorn main:app --host 0.0.0.0 --port 8088`).
- **Источник стакана**: Opera запущена с `--remote-debugging-port=9222`, вкладка `c2c.binance.com` (USDT/IDR) залогинена и держит VPN включённым. `market_watcher.py` подключается к ней через Chrome DevTools Protocol и дёргает публичный `adv/search` из контекста уже авторизованной человеком вкладки — Cloudflare не блокирует. Обновление каждые 4 сек.
- **P&L**: официальный `GET /sapi/v1/c2c/orderMatch/listUserOrderHistory` с обычным (не мерчантским) API-ключом. На Mac прямые TLS-запросы к `api.binance.com` блокируются на уровне сети — поэтому `pnl_remote.py`/`balance_remote.py` выполняются на VPS3 (`root@187.127.114.34`) по SSH, backend дёргает их раз в 90 сек (P&L) и 60 сек (баланс).
- **Подсветка действия** (SELL vs BUY) на дашборде учитывает не только сырой баланс, но и `open_sell_remainder_usdt` (незакрытый долг по докупке после частичной продажи) — иначе путает, если держишь USDT, но на самом деле ещё должен докупить.
- Все суммы USDT на дашборде округлены до целых для удобства выставления заявок.

## Важные находки за сессию

1. **Спред на USDT/IDR Bank Transfer ~0.15-0.21%** — подтверждено живыми данными несколько раз.
2. **Merchant-статус нужен для авто-управления объявлениями через официальный API** — `/sapi/v1/c2c/ads/search` и `/ads/update` дают 401/500 с обычным ключом. Требования мерчанта: 300-1000 реальных P2P-сделок/мес, Verified Plus KYC, депозит в Funding wallet — далеко при текущем темпе.
3. **Видимость объявления критична**: способ оплаты должен включать общий `BANK` (не только конкретные банки) — иначе невидимо большинству фильтрующих по Bank Transfer. Уже исправлено.
4. **Размер лимита тоже критичен**: рынок в основном торгует тикетами 10-50+ млн IDR; у нас лимит ~2.8 млн IDR — структурное ограничение капитала ($300), не баг. Решается увеличением капитала на следующем шаге.
5. **Приватные bapi-эндпоинты** (`c2c/adv/list-by-page`, `c2c/order-match/order-list` — "мои объявления"/"мои ордера" в реальном времени) защищены AWS WAF + CSRF-токеном, который не лежит в куках/localStorage. **Решено не обходить** — та же причина, по которой не стали обходить Cloudflare Turnstile на публичном стакане. Значит и прямое авто-обновление цены через API тоже закрыто без мерчанта.

## Открытый вопрос на следующую сессию

Пользователь хочет **авто-обновление цены/лимита заявки раз в минуту** в зависимости от рынка и остатка баланса. Прямой путь (signed API) закрыт мерчантским гейтом, путь через приватные bapi — закрыт WAF/CSRF.

**Предложенная следующая идея, ещё не реализована**: автоматизировать не HTTP-запросы, а сами клики в браузере через Playwright (зайти в "My Ads" → Edit → поменять цену/лимит → Confirm) — это не обход защиты, потому что реальный клик в реальной странице сам формирует запрос с правильными токенами (как у человека). Риски, которые надо проверить:
- Не потребует ли Binance доп. подтверждения (SMS/email-код) на каждое редактирование цены — если да, полная автоматизация невозможна.
- Нужно тестировать осторожно на реальном объявлении с реальными деньгами — начать под наблюдением пользователя, не давать полную автономность сразу.

Решение пользователя: отложено до следующей сессии ("продолжим завтра").

## Новое: разделить логику лимита SELL vs BUY

Поймали важную asymметрию при разборе лимитов объявлений:
- **SELL-сторона**: верхний лимит можно безопасно ставить = доступному USDT-балансу — он виден через `/api/balance` (бот это реально знает).
- **BUY-сторона**: верхний лимит должен ограничиваться суммой, которая реально лежит **в банке (IDR)** для оплаты — а это банковский счёт, бот туда не достаёт вообще (вне Binance API, нет доступа без банковской интеграции).

**Задача на завтра**: добавить на дашборд ручное поле "Доступно в банке для покупки (IDR)" — бот будет учитывать его при подсказке лимита для BUY-объявления (явно показывать "максимум ≤ это значение"), а не молча считать, что денег в банке бесконечно. Это ручной ввод, не автоматическое чтение (банк не в API) — но рекомендацию по лимиту бот должен строить именно от этого числа, а не от USDT-баланса (тот релевантен только для SELL).

## Масштабирование на $10,000 — что проверить перед разгоном объёма

При расчёте на $10,000/день всплыли два новых потолка, которых не было на $300-500 (где главным ограничением была видимость объявления из-за маленького лимита):
1. **Дневные лимиты P2P по уровню верификации Binance** — не проверены для этого аккаунта, при обороте ~$200k/день (20 циклов × $10k) есть риск упереться или привлечь внимание compliance.
2. **Лимиты на перевод у самих банков** (BCA/Mandiri/Permata — дневные/разовые лимиты перевода) — отдельный потолок сверху, независимый от Binance. Нужно узнать фактические лимиты по используемым счетам пользователя, прежде чем планировать объём на крупном капитале.

Оба пункта — просто **узнать факты** (лимиты Binance по тиру верификации + лимиты конкретных банковских счетов), не требуют разработки, но должны быть проверены до масштабирования капитала к $5,000-10,000+.

## Как поднять всё с нуля после перезагрузки Mac

```bash
# 1. Opera с remote debugging
/Applications/Opera.app/Contents/MacOS/Opera --remote-debugging-port=9222 \
  --user-data-dir="$HOME/Library/Application Support/com.operasoftware.Opera"
# открыть в ней p2p/c2c.binance.com, залогиниться, включить VPN

# 2. Backend
cd ~/Desktop/Check/backend
source venv/bin/activate
uvicorn main:app --host 0.0.0.0 --port 8088
```

Дашборд: `http://localhost:8088`. Учётные данные API-ключа — в `backend/.env` (не в git).
