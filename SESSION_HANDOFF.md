## TL;DR для следующей сессии

Это была огромная сессия — от "сделай дашборд стабильным" до полноценного code review. По порядку, что случилось:

1. **Стабильность инфраструктуры решена** — `launchd` держит backend (KeepAlive, авто-рестарт) и следит за Opera (watchdog раз в 60с, чинит `--remote-debugging-port` если слетел).
2. **Найдена и исправлена критическая дыра в учёте**: Binance P2P берёт ~0.1% комиссии в USDT на КАЖДУЮ сделку (поле `commission` в API, нигде в UI не видно) — `pnl_tracker.py` и весь advisor.py были слепы к ней. День, который выглядел как +Rp 171,278 "прибыли", на самом деле **-Rp 114,195.56 чистого убытка** после комиссии. Исправлено в коде, синкнуто на VPS3.
3. **Построена cost-basis floor защита** в advisor.py — никогда не рекомендовать продавать ниже (своя цена покупки + комиссия за круг).
4. **Полный `/code-review` (high effort)** — 9 находок (7 CONFIRMED, 2 PLAUSIBLE), 4 самых опасных исправлены сразу же (см. ниже). 4 менее критичных остались открытыми.
5. **Несколько раундов "стоп, слишком лихорадочно"** от пользователя — окно браузера перехватывало фокус при каждом действии бота. Решено через `focus_guard.py` (теперь общий refcounted singleton на все CDP-операции) — **подтверждено живьём** (backend реально видит активное окно через `osascript`/System Events, без проблем с правами).
6. **На конец сессии оба тумблера выключены** пользователем (auto-reprice OFF, order-watch OFF) — он хочет "тишины" до след. сессии. Заявка BUY осталась **онлайн**, рынок продолжает её матчить сам по себе.

## Архитектура (актуальная, файл за файлом)

- `market_watcher.py` — стакан + `find_my_ad()` через CDP, персистентная вкладка. Не менялся сегодня.
- `advisor.py` — rule-engine `push→trend→debt→tight→wide→neutral`, ПЛЮС новое состояние **`floor`** поверх всех (см. ниже). `TIGHT_SPREAD_PCT`/`WIDE_SPREAD_PCT` пересчитаны от `ROUND_TRIP_FEE_PCT=0.2` (было 0.12/0.25 — целиком внутри зоны гарантированного убытка, не зная о комиссии).
- `pnl_tracker.py` — `poll()` теперь тащит `commission` из API, `_fifo_match`/`_fifo_annotate` вычитают реальную комиссию обеих ног из `profit_idr`. **Деплоится на VPS3 вручную** (`scp backend/pnl_tracker.py root@187.127.114.34:/root/pnl_tracker.py`) — сегодня синкнуто, актуально.
- `ad_repricer.py` — `AdRepricer.reprice()` теперь принимает `is_pending_fn` (перепроверяет прямо перед кликом Post И перед Confirm — сужает гонку), cooldown ставится сразу после Confirm (не после верификации — флаки таймаут на верификации раньше гонял лишние попытки), но **снимается** при подтверждённом mismatch (раньше блокировал на 120с даже зная что цена не та).
- **`order_watcher.py` — новый файл.** `OrderWatcher` — отдельная CDP-вкладка, читает Processing-таб раз в `ORDER_WATCH_INTERVAL_SECONDS=30`, ловит новые ордера (счётчик 15 минут на оплату), отдаёт через `/api/pending-orders`. `has_pending()` гейтит auto-reprice. **Своя кнопка вкл/выкл** `/api/order-watch` (GET/POST) — **на дашборде кнопки ПОКА НЕТ**, только API. Пользователь просил добавить — TODO №1 ниже.
- **`focus_guard.py` — новый файл.** `FocusGuard` — refcounted singleton (НЕ один на вызов), общий лок+счётчик на все одновременные CDP-операции, чтобы order_watcher и ad_repricer не путали друг друга кто что "восстанавливает". Подтверждено рабочим на деле.
- `main.py` — `auto_reprice_loop()` теперь требует **И** `repricer.enabled` **И** `order_watch_state["enabled"]` **И** `pnl_cache["summary"] is not None` **И** `not order_watcher.has_pending()` — четыре независимых гейта (первые два добавлены сегодня после code review, нашёл что `has_pending()` замораживается если order-watch выключен). `AUTO_REPRICE_INTERVAL_SECONDS=60` (было 30, пользователь попросил пореже). Новые эндпоинты: `/api/order-watch`, `/api/pending-orders`, `_suggested_limits()` встроена в `/api/bank-balance` ответ.
- `static/index.html` — добавлены: подсказка лимитов заявки при Save баланса, баннер+отдельный звук (3 раза) на "заявку взяли — твоя очередь платить".

## Что НЕ доделано из code review (4 находки, не критично для денег)

1. **floor не защищает состояние "wide"** — если позиция глубоко в минусе но спред широкий, защита молчит (recommended_price=None в wide, floor проверяет только не-None).
2. **klaxon может противоречить floor** — "рынок против тебя, действуй!" может звучать одновременно с "держи, не продавай в минус".
3. **Дашборд не показывает что auto-reprice на паузе** из-за открытого ордера — выглядит как будто всё работает нормально.
4. **Риск потерять строку ордера на один цикл опроса** (PLAUSIBLE, самоисправляется через 30с) — `order_watcher.py` фильтрует строки с <5 ячеек, может словить полу-отрендеренную строку.

## Как поднять всё с нуля (launchd теперь делает это сам, но если что-то не так)

```bash
# Backend — должен быть жив всегда (KeepAlive), но если нет:
launchctl kickstart -k gui/$(id -u)/com.check.backend
# Opera watchdog чинит флаг сам раз в 60с, но форсировать вручную:
launchctl kickstart -k gui/$(id -u)/com.check.opera-watchdog

# Проверить всё:
curl -s http://localhost:9222/json/version            # Opera CDP
curl -s http://localhost:8088/api/orderbook | python3 -m json.tool  # backend + стакан
launchctl list | grep com.check                        # статусы launchd-юнитов
```

После рестарта backend **сбрасывается** (нужно переустановить руками через дашборд/API): `auto-reprice` (OFF), `order-watch` (по коду default ON, но проверь — могло поменяться), `active-side override` (auto), `bank_balance` (дефолт 7M).

Ключи API — `backend/.env` (не в git). После правок `pnl_tracker.py` — синкать на VPS3: `scp backend/pnl_tracker.py root@187.127.114.34:/root/pnl_tracker.py`, и перезапускать ничего на VPS не нужно (`pnl_remote.py` запускается заново при каждом SSH-вызове из `pnl_refresh_loop`).

launchd plist'ы лежат в `~/Library/LaunchAgents/com.check.backend.plist` и `~/Library/LaunchAgents/com.check.opera-watchdog.plist` — оба указывают на `~/Projects/Check`, не трогать руками без причины.
